Deep Dive Into Features

How each check works, when you use it, and why it matters

📋

INPUT_CONTRACT

Request schema validation & testing

How It's Used
schema: type: object required: [prompt] properties: prompt: type: string minLength: 1 maxLength: 1000
Real Uses
Validates JSON Schema syntax
Tests valid samples pass
Tests invalid samples fail
Prevents malformed requests
Real Impact
Ensures your agent receives well-formed requests. Prevents crashes from unexpected input types. Catches schema errors before production.
🛡️

FALLBACK_DECLARED

Operational safety & incident response

How It's Used
fallback_declared: kill_switch: type: feature_flag name: disable_agent ownership: team: platform-eng oncall: oncall-email runbook_url: https://wiki/runbook
Real Uses
Declares kill switch mechanism
Specifies fallback behavior
Assigns team ownership
Links incident runbook
Real Impact
Ensures you can kill the agent in <5 seconds. Team knows who to page. Incident response is documented. Production outages are handled fast.
🔐

IDENTITY_BOUNDARY (v0.2)

Access control & rate limiting

How It's Used
identity_boundary: authentication: required rate_limit: 100 data_isolation: - user_owned_only - no_cross_access
Real Uses
Requires user authentication
Limits requests per user/hour
Isolates customer data
Prevents unauthorized access
Real Impact
Stops attackers from free agent access. Prevents customers from seeing each other's data. Blocks abuse and brute force. Makes multi-tenant safe.
💰

ACTION_BUDGET (v0.2)

Resource & cost controls

How It's Used
action_budget: max_tokens_per_req: 5000 max_retries: 3 max_daily_cost: 1000 max_concurrent: 10
Real Uses
Caps tokens per request
Limits retry attempts
Sets daily spending limit
Bounds concurrent load
Real Impact
Prevents infinite loop $50K bills. Stops resource exhaustion crashes. Ensures predictable monthly costs. Protects service availability.
