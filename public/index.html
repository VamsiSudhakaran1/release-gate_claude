Why This Is Necessary

You have QA, UAT, integration tests. Why add governance enforcement?

🤖 AI Agents Are Different

Traditional QA tests behavior. But AI agents can behave unpredictably in production. QA validates "does it work?" Governance validates "is it safe to run?"

⚠️ QA Tests Functionality

QA verifies agent returns correct answer. But QA can't test:

  • • Did you set cost limits?
  • • Is authentication required?
  • • Do you have a kill switch?
  • • Who owns this agent?

🔐 Governance is Compliance

QA passes → Deploy to production. But compliance teams want proof:

  • • Cost constraints documented
  • • Access control enforced
  • • Rollback plan exists
  • • Audit trail complete

📊 Automated Evidence

release-gate creates an audit trail. Instead of "we followed best practices," you have "agent passed all 4 governance checks on 2026-03-18 at 14:32 UTC."

⏱️ 10x Faster Deployments

Manual governance review = 1-3 days. release-gate checks in 7 seconds. Developers know instantly if they're compliant. No bottlenecks.

🛡️ Prevents Costly Failures

Agent deployed without rate limits → Costs $50k in 2 hours. release-gate catches this before deployment. Policy enforcement saves money.

The 4 Governance Checks

What release-gate validates before deployment

📋

INPUT_CONTRACT

Validates that request schemas are explicit and tested.

Validates:
✓ Schema JSON defined
✓ Valid samples pass
✓ Invalid samples fail
Why: Prevents agents from accepting garbage input that could cause failures.
⏹️

FALLBACK_DECLARED

Confirms operational safeguards are in place.

Validates:
✓ Kill switch defined
✓ Fallback mode exists
✓ Team ownership clear
✓ Runbook provided
Why: Ensures you can stop the agent if something goes wrong.
🔐

IDENTITY_BOUNDARY

Enforces access control and rate limiting.

Validates:
✓ Authentication required
✓ Rate limits configured
✓ Data isolation defined
Why: Prevents unauthorized access and abuse.
💰

ACTION_BUDGET

Validates resource constraints are defined.

Validates:
✓ Max tokens per request
✓ Max retries configured
✓ Daily cost limit set
✓ Concurrent requests capped
Why: Prevents runaway costs and resource exhaustion.
