Why release-gate?

⚡ One Command Setup

Run `release-gate init` and answer 6 questions. Complete governance in 5 minutes. No manual configuration.

💰 Budget Simulation

Project realistic costs before deployment. Accounts for retries, caching, and peak usage. Works with 10+ models.

⚡ Policy Engine

Define what's critical (blocks deployment) vs. flexible (needs review). Zero infrastructure. Runs in CI/CD.

🔒 Access Control

Enforce authentication, rate limiting, and data isolation. Identity boundaries that stick.

🛡️ Safety First

Require fallback modes, kill switches, and runbooks. Never deploy without an escape hatch.

✅ Contract Validation

Validate input schemas against samples. Catch configuration errors early.

CI/CD Platform Support

GitHub Actions

Init creates `.github/workflows/release-gate.yml` with full integration. Run on every push and PR.

GitLab CI

Init creates `.gitlab-ci.yml` with governance stage. Integrated with merge request workflows.

Jenkins

Init creates `Jenkinsfile` pipeline. Works with existing Jenkins infrastructure.

Manual / Other

Just creates `governance.yaml`. You handle CI/CD integration yourself.

The 5 Governance Checks

💵 ACTION_BUDGET

Prevents cost explosions. Set max daily cost and block deployment if exceeded.

🎯 BUDGET_SIMULATION

Project costs realistically. Accounts for retries, caching, spiky usage. Daily/monthly/annual estimates.

🛑 FALLBACK_DECLARED

Ensure safety measures. Require fallback mode, kill switch, team owner, and runbook.

🔑 IDENTITY_BOUNDARY

Enforce access control. Require authentication, rate limiting, and data isolation.

📋 INPUT_CONTRACT

Validate schemas. Define required inputs and validate against samples.

Running Validation

Local Testing

release-gate run governance.yaml # Exit codes: # 0 = PASS (deploy) # 10 = WARN (review needed) # 1 = FAIL (fix first)
